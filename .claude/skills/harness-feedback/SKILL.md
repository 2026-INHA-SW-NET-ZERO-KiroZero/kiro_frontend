# harness-feedback — 피드백 루프 / 교정 지시 자동 진화

반복되는 실수를 잡아 하네스 규칙으로 승격시킨다. 사람이 매번 지적하지 않아도 다음부터 자동으로 막히게 한다.

## 트리거

- 사용자가 "같은 실수 반복돼", "규칙 추가해줘" 등으로 요청.
- kiro-qa가 **동일 패턴 위반을 2회 이상** 감지.
- **Hook 자동 교정에서 같은 규칙이 3회 이상** 걸림.

## 워크플로우

1. **반복 문제 패턴 파악** — 어떤 위반이, 어디서, 몇 번 발생했는지 정리한다.

2. **4단계 보강 제안** (낮은 단계부터):

   **Level 1 — 문서 보강 (소프트 강제)**
   `docs/FRONTEND.md` 또는 `docs/DESIGN.md`에 규칙을 추가한다.

   **Level 2 — 린트 규칙 추가 (자동 교정 강화)**
   ESLint 규칙으로 변환해 `eslint.config.js`에 추가 → Hook이 자동으로 잡는다.

   **Level 2.5 — 린트 교정 지시 업데이트**
   에이전트가 특정 린트 에러를 **잘못된 방식으로 반복 수정**할 때(교정 지시가 불충분하다는 뜻):
   - 실제로 어떻게 잘못 고쳤는지 패턴 분석
   - 올바른 교정 방법을 반영해 규칙의 `message`(교정 지시)를 업데이트
   - 예: `any` → 매번 `as unknown as X`로 우회 → 교정 지시를 "unknown + 타입 가드 사용, `as unknown as X` 캐스팅 금지. 예: `if (isUser(data)) { data.name }`"로 개선

   **Level 3 — 구조적 테스트 추가 (기계적 검증)**
   `__tests__/structural/`에 패턴 검증 테스트를 만든다(예: 모든 Screen이 `SafeAreaView`를 쓰는지, 하드코딩 색이 없는지).

3. **사용자 확인 후 적용** — 어느 레벨로 갈지 제안하고, 승인받아 적용한다.

4. **기록** — `docs/design-docs/feedback-log.md`에 날짜·문제·레벨·변경 내용·관련 이슈를 남긴다.

## 레벨 선택 기준

- 빈도 낮음·주관적 → Level 1(문서). 빈도 높음·기계 판별 가능 → Level 2/3. 잘못 고치는 패턴 → Level 2.5.
- 과하게 강제하면 거짓양성으로 작업이 막힌다. 가장 낮은 효과적 단계를 고른다.
