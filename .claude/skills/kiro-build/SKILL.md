# kiro-build — 화면/기능 구현 오케스트레이터

KiroZero의 화면·기능을 **에이전트 팀**으로 구현하는 워크플로우. "누가 언제 어떤 순서로 협업하는가"를 정의한다.

## 실행 모드: 에이전트 팀 (기본)

리더(나)가 팀을 구성하고 작업을 할당하면, 팀원들이 `SendMessage`로 직접 조율한다.

- **kiro-ui** — 화면/컴포넌트 (`model: opus`)
- **kiro-logic** — 상태/데이터/로직 (`model: opus`)
- **kiro-qa** — 검증 (`general-purpose` 타입, `model: opus`, 별도 컨텍스트)

> 모든 Agent 호출에 `model: "opus"`를 명시한다. 하네스 품질은 추론 능력에 직결된다.

## 워크플로우

### 1. 작업 분석

- 대상 화면/기능과 PRD 섹션(§3.x, §4, §5)을 확정한다.
- 기반 작업(디자인 토큰·공용 컴포넌트·네비게이션·타입)이 선행돼야 하면 먼저 처리한다.
- GitHub 이슈 기반이면 `docs/exec-plans/`의 작업 계획을 참조한다.

### 2. 작업 분해 + 할당 (파이프라인 패턴)

대부분의 화면은 **로직 → UI → 검증** 순서로 의존한다.

- `TaskCreate`로 작업을 만들고 의존성을 건다: `kiro-logic`(타입·훅·시드) → `kiro-ui`(화면·컴포넌트) → `kiro-qa`(검증).
- 독립적인 공용 컴포넌트는 병렬로 진행할 수 있다.

### 3. 자체 조율

- kiro-ui ↔ kiro-logic은 훅 시그니처/반환 shape을 `SendMessage`로 직접 맞춘다.
- 리더는 진행 상황을 모니터링하고 차단 이슈만 개입한다.

### 4. 점진적 검증

- 각 화면/모듈 완성 직후 kiro-qa가 검증한다(전체 완성 후 1회 금지).
- 반려 → 담당 에이전트 수정 → 재검증. **최대 2회 재시도**, 초과 시 사람에게 보고.

### 5. 종합 + 정리

- 결과를 모으고 `docs/generated/component-inventory.md`를 갱신한다.
- 팀을 정리한다. 다음 Phase에 다른 조합이 필요하면 산출물을 파일로 남기고 재구성한다.

## 데이터 전달 프로토콜

- **태스크 기반**(조율): `TaskCreate`/`TaskUpdate`로 진행·의존 관리.
- **메시지 기반**(실시간): `SendMessage`로 시그니처 협의·반려 피드백.
- **파일 기반**(산출물): 코드는 `src/` 정해진 경로에, 중간 산출물은 `docs/exec-plans/`에.

## 에러 핸들링

- Hook 자동 교정으로 못 잡는 고수준 위반은 kiro-qa가 잡는다.
- 1회 재시도 후 재실패 시 해당 결과 없이 진행하고 보고서에 누락을 명시한다.
- 상충 데이터는 삭제하지 않고 출처를 병기한다.
- 같은 패턴 위반 2회+ → harness-feedback 연계.

## 기준 패턴 (반드시 따를 것)

- 디자인 토큰: `src/theme/theme.ts`만 사용. 하드코딩 색 금지.
- 폴더: 도메인(feature) 기반. `src/app/`=라우트, `src/features/{도메인}/`=화면·훅, `src/components/`=공용.
- 상태: Zustand 도메인 스토어 + 화면-로컬 useState. 폼은 useState 직접 관리.
- 더미데이터는 커스텀 훅으로 감싼다(API 교체 대비).
- 라이트 모드 전용. 한국어 카피는 PRD 문자열 그대로.
- 상세 규칙: `docs/FRONTEND.md`, `docs/DESIGN.md`.

## 범위 밖

- 백엔드 API/서버/DB 구현. 백엔드 라벨 작업은 거부한다.

## 테스트 시나리오

- **정상 흐름:** "로그인 화면 만들어줘" → 리더가 kiro-logic(authStore·검증 유틸) → kiro-ui(Login 화면) → kiro-qa(검증) 파이프라인 구성 → QA 통과 → inventory 갱신.
- **에러 흐름:** kiro-ui가 하드코딩 색 사용 → Hook은 못 잡음 → kiro-qa가 critical 반려 → kiro-ui 토큰으로 수정 → 재검증 통과. 2회 반려 후에도 실패 시 사람에게 보고.
