## 목표

로그인·회원가입 두 화면을 디자인과 픽셀 동일하게 구현하고, 백엔드 `auth-controller`와 연동한다. 인증의 단일 진실원천은 **JWT 토큰** — 응답 `token` 유무로만 로그인 상태를 판정한다. 마이페이지에 로그아웃 버튼을 추가한다.

## 주요 결정사항 (확정)

### 결정 1 — 인증 상태: 기존 AuthContext + SecureStore 재사용 ✅ **확정(사용자 컨펌)**

Zustand는 쓰지 않는다. `#22`의 **AuthContext + SecureStore**(`src/hooks/useAuth.ts`의 `login(token)`/`logout()`/`authed`, `src/lib/tokenStorage.ts` key `kiro_auth_token`)가 token 단일 판정·영속화를 이미 충족한다. 새 스토어를 만들지 않고 이를 재사용한다.
→ API 호출 훅(`useAuthApi`)이 응답 `token`을 받아 `useAuth().login(token)`에 넘긴다. 로그아웃은 `useAuth().logout()`.

### 결정 2 — API 연동: Swagger 대조 완료 → **UI + API 동시 구현** ✅ **게이트 해소**

`#23`이 Swagger를 추출해 `api-schema.md`(22개 EP + DTO)와 `backend-swagger.json` 스냅샷을 동기화했다. 이슈 명세와 항목별 대조 결과 **완전 일치**:

- `SignupRequest { email*, password*, nickname*, cookingSkill*: CookingSkill, allergyTags: string[] }`
- `LoginRequest { email*, password* }`
- `AuthResponse { userId: long, email, nickname, cookingSkill, allergyTags, token }`
- `CurrentUserResponse` = AuthResponse − token
- `bearerAuth` HTTP Bearer(`opaque-token`), 보호 자원에 `Authorization: Bearer <token>`
  → 추측 연동 아님. Phase 0의 "Swagger 대조·명세 갱신"은 #23으로 **완료**. 남은 건 타입 정의 + allergy-tags 실데이터 확인뿐.
  → 비밀번호 `min 8`은 이슈에 명시(클라 검증으로 적용). DTO 표기엔 길이 제약이 안 보이므로 `backend-swagger.json`에서 한 번 확인.

### 결정 3 — 알레르기 칩 라벨 ↔ 서버 태그 코드: 구조 확정, 정적 하드코딩 금지 ✅

`AllergyTagListResponse { allergyTags: AllergyTagItemResponse[] }`, `AllergyTagItemResponse { tag, labelKo, description }`.
→ 칩 라벨 = `labelKo`, 서버로 보낼 코드 = `tag`. 칩 10종을 **정적으로 박지 말고 `GET /api/v1/allergy-tags`로 받아 렌더**(라벨↔코드 매핑을 서버가 진실원천). 이모지(🥜 등)는 디자인 핸드오프(`KiroZero.dc.html`)와 `tag` 기준으로 매핑. 실제 `tag` 값 10종은 구현 시 GET 응답으로 확정.
