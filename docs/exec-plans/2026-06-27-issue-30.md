## API ↔ UI 타입 불일치 분석

### MenuCandidateResponse vs MenuCandidate

| API 필드                                    | UI 필드        | 매핑 방식                                             |
| ------------------------------------------- | -------------- | ----------------------------------------------------- |
| `menuName`                                  | `name`         | 직접                                                  |
| `menuType`                                  | `type`         | `'저탄소'`\|`'일반'` — `menuType` 값에 따라 매핑 필요 |
| `cookingTimeMinutes: int`                   | `time: string` | `${n}분` 변환                                         |
| `difficulty`                                | `diff`         | 직접                                                  |
| `recommendationReason`                      | `desc`         | 직접                                                  |
| `usedLeftoverIngredients[].nameKo`          | `needed[].n`   | 직접                                                  |
| `usedLeftoverIngredients[].plannedUseGrams` | `needed[].q`   | `${n}g` 변환                                          |
| `purchaseItems[].name`                      | `missing[].n`  | 직접                                                  |
| `purchaseItems[].quantityGrams`             | `missing[].q`  | `${n}g` 변환                                          |
| ❌ 없음                                     | `emoji`        | **optional** — fallback `'🍽️'`                        |
| ❌ 없음                                     | `score`        | **optional** — 화면은 rank 인덱스로 이미 처리         |
| ❌ 없음                                     | `co2`          | **optional** — 화면에서 조건부 렌더링                 |

### SessionParticipantStatusResponse vs PartyProfile

| API 필드                        | UI 필드                  | 매핑 방식                              |
| ------------------------------- | ------------------------ | -------------------------------------- |
| `cookingSkill: CookingSkill`    | `skill: string`          | `HIGH→'상'`, `MEDIUM→'중'`, `LOW→'하'` |
| `allergyTags: string[]`         | `allergies: string[]`    | 직접                                   |
| `ingredients[].nameKo`, `count` | `bring[].e`, `bring[].n` | emoji 없음 → `''`, `nameKo count개`    |
| `canPurchase: boolean`          | `extra: boolean`         | 직접                                   |

### VoteMenu (LatestRecommendationResponse 기반)

- `voteSummary: { [candidateLabel]: long }` → `VoteMenu.votes` 매핑
- `emoji`, `co2` → **optional** 처리

### DecidedMenu (selectedMenu 기반)

- `SelectedMenuSummaryResponse { candidateLabel, menuName, menuType }` 만 존재
- `recipe`, `roles`, `time`, `servings` 등 → **이슈 범위 밖** (cooking-guide API 별도)
- 이 이슈에서 `useDecidedMenu`는 **selectedMenu 요약만 반환**, 나머지는 빈 배열로 fallback

- [ ] **1. `src/types/session.ts` 생성** — API 원본 타입 (서버 응답 shape 그대로)
  - `SessionStatusResponse`, `SessionParticipantStatusResponse`, `SessionIngredientStatusResponse`
  - `UpdateSessionIngredientsRequest`, `UpdateSessionIngredientsResponse`
  - `RecommendationResponse`, `LatestRecommendationResponse`
  - `MenuCandidateResponse`, `CandidateUsedIngredientResponse`, `PurchaseItemResponse`
  - `MenuVoteRequest`, `MenuVoteResponse`
  - `SelectedMenuSummaryResponse`
  - enum `VoteType` = `'A'|'B'|'C'|'D'|'E'`

- [ ] **2. `src/types/menu.ts` 수정** — `emoji?: string`, `score?: number`, `co2?: string` optional
  - `MenuCandidate.emoji`, `score`, `co2` → `?` 추가
  - `VoteMenu.emoji`, `co2` → `?` 추가
  - `DecidedMenu.recipe`, `roles` → 이미 배열이므로 변경 불필요

- [ ] **3. `src/lib/sessionApi.ts` 생성** — 세션 도메인 API 클라이언트
  - `getSession(slotId: number)` → `GET /api/v1/sessions/{slotId}`
  - `updateSessionIngredients(slotId, body)` → `PUT /api/v1/sessions/{slotId}/ingredients`
  - `requestRecommendation(slotId)` → `POST /api/v1/sessions/{slotId}/recommendations`
  - `getLatestRecommendation(slotId)` → `GET /api/v1/sessions/{slotId}/recommendations/latest`
  - `submitVote(slotId, body)` → `POST /api/v1/sessions/{slotId}/votes`

- [ ] **4. `usePartyPool(slotId)` 교체** — `getSession` → participants 매핑 → `PartyProfile[]`
  - 훅 시그니처: `usePartyPool(slotId: number): AsyncResult<PartyProfile[]>`
  - CookingSkill enum → '상'/'중'/'하' 변환
  - ingredients → `bring[]` (emoji `''`, `n = nameKo + ' ' + count + '개'`)

- [ ] **5. `useMenuCandidates(slotId)` 교체** — `getLatestRecommendation` → candidates 매핑
  - 훅 시그니처: `useMenuCandidates(slotId: number): AsyncResult<MenuCandidate[]>`
  - `cookingTimeMinutes` → `${n}분`, `plannedUseGrams` → `${n}g`

- [ ] **6. `useVoteMenus(slotId)` 교체** — `getLatestRecommendation` → candidates + voteSummary 매핑
  - `voteSummary[candidateLabel]` → `votes` 매핑
  - emoji, co2 없음 → optional 처리

- [ ] **7. `useDecidedMenu(slotId)` 교체** — `getLatestRecommendation` → selectedMenu 매핑
  - `selectedMenu.menuName`, `menuType` → 최소 필드만 채우고 나머지 빈 배열 fallback

- [ ] **8. `submitVote` 뮤테이션 훅** (`useSubmitVote`) 추가 — MyApplicationScreen에서 투표 제출 시 사용
  - `MyApplicationScreen`의 `submitVote` 함수가 현재 로컬 state만 변경 → API 호출로 교체

- [ ] **8-1. 투표 확정 폴링 훅** (`useVoteConfirmed`) 추가
  - **백엔드 제약**: 내가 투표를 제출해도 다른 인원이 아직 투표 중이면 `MenuVoteResponse.confirmed = false` → 서버푸시 없음, **폴링으로만 확인 가능**

  - `MyApplicationScreen`에서 투표 제출 후(`votingDone = true`) 폴링 시작 → 확정되면 `result` stage 전환
  - 화면 언마운트 시 폴링 정리 (clearInterval)

- [ ] **9. 더미데이터 제거**
  - `src/data/party.ts` 파일 삭제
