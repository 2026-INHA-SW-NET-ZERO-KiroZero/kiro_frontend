# Core Beliefs — KiroZero 핵심 설계 원칙

> 이 문서의 역할: KiroZero가 코드/디자인에서 지키는 근본 철학. 세부 규칙(FRONTEND/DESIGN)의 "왜"에 해당한다.
> 디자인 핸드오프(FRONTEND_PRD.md, theme.ts, HTML 프로토타입) 분석에서 도출됐다.

## 1. 디자인은 이미 확정됐다 — 코드는 충실한 재현이다

색·타이포·간격·카피가 hifi로 확정됐다(`theme.ts`, PRD). 코드는 창작이 아니라 **픽셀 충실한 번역**이다.
→ 토큰 단일 출처를 강제하고, 임의 색/문구를 금지한다.

## 2. 단일 출처(Single Source of Truth)

디자인 토큰은 `theme.ts` 하나, 데이터 shape은 `src/types`·`src/data` 하나, 파생 계산은 `src/lib/derive.ts` 하나.
→ 같은 값이 여러 곳에 흩어지면 AI가 옛 패턴을 베껴 드리프트가 스노우볼된다. 출처를 하나로 묶어 막는다.

## 3. 더미데이터와 UI를 분리한다 — 백엔드 교체를 전제로

지금은 클라이언트 시드지만 곧 API로 바뀐다(PRD §5). UI가 시드를 직접 알면 교체가 지옥이 된다.
→ 커스텀 훅이 경계가 된다. UI는 훅만 알고, 훅 내부만 나중에 교체한다.

## 4. 타입은 UI를 구동한다

PRD가 데이터 shape으로 화면을 정의했다. 타입이 정확하면 경계면 버그(필드 누락·이름 불일치)가 사라진다.
→ `any` 금지, shape 우선, 경계면 교차 검증(QA의 핵심).

## 5. 절제(No AI-slop)

장식 그라데이션·랜덤 이모지·과한 모션을 쓰지 않는다. 이모지는 데이터일 때만, 그라데이션은 hero/stat 카드만.
→ 디자인이 의도한 차분한 톤을 지킨다.

## 6. 자동화로 일관성을 지킨다 — 사람의 기억에 의존하지 않는다

저장 시 자동 교정(Hook) → QA 교차 검증 → 반복 시 규칙 승격(harness-feedback) → 주기 청소(/gc).
→ 규칙을 "문서에 적어두는 것"이 아니라 "기계가 강제하는 것"으로 만든다.
